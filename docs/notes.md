# HashQuery 技術メモ

設計判断や内部動作の背景など、仕様書・テスト仕様書には書かない技術的な注記をまとめる。

---

## `as` 変数と tie の関係

`as $tbl` で指定した変数は、`where` / `having` ブロックの実行中に `HashQuery::RowHash` インスタンスとして tie される。これにより `$tbl->{col}` のようなハッシュアクセスが `HashQuery::Value` を返す値オブジェクトとして機能する。

この tie は `where` / `having` の評価中のみ有効であればよいが、同じ変数を複数のクエリで使い回すと、2回目のクエリ開始時点でその変数がすでに tie 済みの状態になっており、正しく動作しない問題が生じうる。

この問題を解決するため、クエリ完了後に `as` 変数へプレーンなハッシュリファレンス（`{ count => N, affect => M }`）を上書きする設計にしている。プレーンなハッシュリファレンスを代入することで tie が解除され、次のクエリで同じ変数を `as` に渡しても安全に動作する。

`as` 変数をクエリ間で使い回す場合は `our`（パッケージ変数）として宣言する必要がある。`my` 変数では tie の仕組みが正しく動作しない。

---

## `SELECT` / `DELETE` を大文字にしている理由

Perl の組み込み `select`（IO セレクタ）および `delete`（ハッシュ要素削除演算子）は、パーサーレベルで特別扱いされる演算子であり、ユーザー定義サブルーチンのプロトタイプ定義でオーバーライドできない。

`select` については実際に `sub select (;$)` が機能するが、`delete` は Perl のパーサーが構文として先に解釈するため、同名のサブルーチンをエクスポートしてもユーザーコード側で組み込みが呼ばれてしまう。両者を統一した形で提供するため、DSL 関数名を `SELECT` / `DELETE` （大文字）に統一した。

内部のハッシュキー（`{ select => ... }`、`{ delete => 1 }`）はモジュール内部のデータ構造であり、大文字化の対象外。

---

## `SELECT` は省略可能（内部仕様）

`SELECT` DSL は省略可能。`DELETE` も `UPDATE` も指定されなかった場合、暗黙的に SELECT モードとして動作する。`SELECT` を明示した場合は列射影の指定（配列リファレンスまたは `except`）としてのみ機能する。

`DELETE` と `UPDATE` の同時指定は die するが、`SELECT` と `DELETE` / `UPDATE` の同時指定は die しない（`SELECT` によって指定された列リストは DELETE / UPDATE モードでは無視される）。

---

## `_row` 内部カラムについて

`query` 実行時に各行へ自動付加する 0 始まりの行番号カラム。`grep_concat` が前後行を参照する際の基点として使用する。最終出力には含まれない。

`select` モードでは列射影ステップで除外される。`delete` モードでは列射影を行わないが、AOH を返す前に明示的に除外する。
