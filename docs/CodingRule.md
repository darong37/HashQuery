# Coding Rule

## 1. 変数名は小文字を基本とする

- Perl の慣習に合わせ、変数名は原則として小文字で書く
- 例外は `asNull` のように、意味上必要な最小限の大文字だけにとどめる
- 無意味な大文字始まりや全大文字のローカル変数は使わない

## 2. 配列・配列相当は複数形にする

- `@dsls`, `@rows`, `@cols` のように、複数要素を持つものは複数形で表す
- 単数と複数が見分けやすい名前にする
- `%` や `@` が付いていても、意味として単数・複数が曖昧な名前は避ける

## 3. 抽象的すぎる変数名を使わない

- `ref`, `part`, `data`, `item`, `obj` のような汎用語は原則避ける
- その値が何者かを直接表す名前を使う
- 例: `ref` ではなく `alias`、`part` ではなく `dsl`

## 4. 名前は「役割」を表し、「実装属性」を表しすぎない

- 名前はその値の役割を表すものにする
- 例: alias を表す値なら `alias` と書く
- `alias_ref` のように、型情報を名前へ過剰に埋め込まない
- 型が必要ならコードを読めば分かる形にする

## 5. 長すぎる説明調の変数名を避ける

- `output_columns`, `working_table`, `select_node` のような業務アプリ的に長い名前は避ける
- Perl らしく、短くても役割が伝わる名前を選ぶ
- 例: `cols`, `tbl`, `sel`, `whr`, `hvg`

## 6. ただし、短ければ何でもよいわけではない

- 短さよりも文脈との一致を優先する
- DSL を扱う場所では `dsl`, `dsls` を使い、`part` のような無関係な一般語にしない
- 略称は、その文脈で自然に読めるものだけ使う

## 7. Perl らしく、直接的で直感的な設計を優先する

- 過度な抽象化や設計パターンを持ち込まない
- `Context -> View` のような多段構造より、直接使うものをそのまま扱う設計を優先する
- 必要以上の内部関数や補助オブジェクトを増やさない

## 8. 小さな処理は小さく書く

- 単純なフィルタ処理は `grep` を優先する
- 単純な変換は `map` やその場の処理で完結させる
- 小さな処理のためだけに補助関数を増やさない

## 9. 仕様で意味が決まっている名前は、そのまま使う

- `dsl`, `alias`, `select`, `except`, `where`, `having` など、仕様上の用語を優先する
- コード側で勝手に別名へ言い換えない
- 仕様書とコードの語彙を一致させる

## 10. Perl に合わない設計パターンを持ち込まない

- 見た目だけ整った抽象化や、一般論としての設計パターンを優先しない
- Perl の強みである、軽さ・直感性・直接性を優先する
- 「かっこよさ」より「読んですぐ分かること」を重視する
