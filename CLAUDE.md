# コーディング規約

## 変数名

- 変数名は小文字を基本とする
- 複数要素は複数形にする（`@dsls`, `@rows`, `@cols`）
- `ref`, `part`, `data`, `item`, `obj` のような抽象的な汎用語は使わない。その値が何者かを直接表す名前を使う
- 型情報を名前に埋め込まない（`alias_ref` ではなく `alias`）
- 長い説明調の名前は避け、短く役割が伝わる名前にする
- 短さより文脈との一致を優先する

## 仕様用語

- 仕様上の用語はそのままコードに使う
- コード側で勝手に別名へ言い換えない
- 仕様書とコードの語彙を一致させる

## 設計方針

- 過度な抽象化や設計パターンを持ち込まない
- 単純なフィルタは `grep`、単純な変換は `map` で完結させる
- 小さな処理のためだけに補助関数を増やさない
- 軽さ・直感性・直接性を優先する

---

# Git 運用

- `git add` はファイル個別指定ではなくプロジェクトディレクトリ単位で行う
- `git status` で未追跡ファイルがないことを確認してからコミットする

---

# 変更フロー

変更は以下の順で行う:

1. 仕様書を更新する
2. テスト仕様書を更新する
3. コードを修正する
4. テストを更新・実行する
5. コミット・プッシュする

---

# 実装前の確認

- 変更を加える前に変更箇所を提示し、承認を得てから実施する
- 変更規模が大きい場合は特に、差分の概要を先に示す

---

# ドキュメント同期

- コード変更時は仕様書・テスト仕様書・README を必ず追随して更新する
- ドキュメントとコードの内容が乖離しないようにする

---

# コミット前の README 確認

コミットを行う前に必ず README.md を確認する:

1. README.md と各種仕様・実装内容に矛盾がないか確認する
2. 追加・更新すべき内容が README.md に反映されているか確認する
3. 不整合や不足があれば修正してからコミットする
