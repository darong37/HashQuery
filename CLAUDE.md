# コーディング規約

- 変数名は小文字基本、役割を直接表す名前を使う（`ref` ではなく `alias`、`part` ではなく `dsl`）
- 複数要素は複数形（`@dsls`、`@rows`、`@cols`）
- 仕様上の用語をそのまま使う（`alias`、`select`、`where`、`having`）
- 短さより文脈との一致を優先する
- 過度な抽象化・設計パターンを持ち込まない。単純なフィルタは `grep`、変換は `map` で完結させる
- 詳細は [docs/CodingRule.md](docs/CodingRule.md) を参照

# Git 運用

- 新しい変更を始める前に必ずブランチを作成し、そのブランチで作業する
- ブランチ名は `feature/<機能名>` または `fix/<修正内容>` の形式とする
- `git add` はファイル個別指定ではなくプロジェクトディレクトリ単位で行う
- `git status` で未追跡ファイルがないことを確認してからコミットする

# 変更フロー

1. 仕様書（docs/spec.md）を更新する
2. テスト仕様書（docs/test-spec.md）を更新する
3. コードを修正する
4. テストを更新・実行する
5. コミットする
6. プッシュ前に `CHANGELOG.md` を日本語で更新する（新機能・変更・修正・ドキュメントの区分で記載）
7. プッシュする

# 実装前の確認

- 変更を加える前に変更箇所を提示し、承認を得てから実施する
- 変更規模が大きい場合は特に、差分の概要を先に示す

# ドキュメント同期

- コード変更時は docs/spec.md・docs/test-spec.md・README.md を必ず追随して更新する
- ドキュメントとコードの内容が乖離しないようにする

# コミット前のソース差分確認

ソースコード（`src/` 以下）を変更するコミットの前に、必ず `git diff` の全体差分をユーザーに提示し、意図通りの変更であることを確認してからコミットする。

# コミット前の README 確認

1. README.md と各種仕様・実装内容に矛盾がないか確認する
2. 追加・更新すべき内容が README.md に反映されているか確認する
3. 不整合や不足があれば修正してからコミットする
